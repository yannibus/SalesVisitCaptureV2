# Sales Visit Capture V2

> AI-powered sales visit preparation, capture, and action plan materialization for Salesforce.
> Built for **Solution Engineers**, **presales**, and **field sales reps**.

Turn every sales meeting into a structured CRM artifact — before, during, and after the visit — with Einstein GenAI doing the heavy lifting.

---

## ✨ What's inside

**Three AI chained experiences, one sales motion:**

| Stage | Prompt | Output |
|---|---|---|
| **Before** — Prep the demo | `Prep_Demo_Grounding` | 5 grounding fields (company profile, pain points, hooks, objections, success criteria) |
| **Before** — Prep the visit | `Prepare_Visit_Brief` | A tactical HTML brief scannable in 45 seconds (context, talking points, opening hooks, objections, success criteria) |
| **After** — Capture the visit | `Generate_Visit_Report` | A HTML executive summary + a structured action plan (max 10 tasks & events) |

Plus a **LWC-based cockpit** (KPIs, action queues, analytics) and an **immersive 3-tab Visit Workspace** (Preparation → Execution → Outcome) that replaces the standard record page.

---

## 🎯 Architecture

### 🅰️ Component A — Demo prep (`Demo_Context__c`)

A lightweight object that acts as the **brief sheet for an upcoming demo or meeting**. The SE enters the target account, meeting type, goal, personas, and a few notes. One click on the Quick Action **"Generate Grounding"** triggers `Prep_Demo_Grounding` (Einstein Flex, GPT-4 Omni, JSON output) which writes 5 grounded AI fields:

- **Company Profile** — market/positioning/strategic stakes
- **Pain Points** — likely industry pains
- **Conversation Hooks** — high-impact opening angles
- **Objection Map** — anticipated objections + counter-arguments
- **Success Criteria** — measurable demo success signals

The grounding is reused as **context** by Component B and by the Visit Brief prompt.

### 🅱️ Component B — Visit capture (`Visit_Capture__c`)

The central record of a sales visit. It combines:

- **Lookups**: VideoCall (In-Person Meeting / ECI), Demo_Context (prepared grounding), Account, Contact, Opportunity, and a soft-link to the source Event
- **Capture fields**: voice transcript (manual or dictation), free notes, file attachments
- **AI fields**:
  - `AI_Visit_Brief__c` (Rich Text) — pre-meeting tactical brief generated by `Prepare_Visit_Brief`
  - `AI_Report__c` (Rich Text) — HTML post-meeting executive report
  - `AI_Action_Plan__c` (JSON) — structured plan (mix of tasks & events)
  - `Detected_Language__c` — auto-detected language of the content

Three AI actions on the record:

- **Generate visit brief** (tab Preparation) — sharpens the rep before the meeting
- **Generate Report** (tab Execution) — transforms transcript + notes + grounding into a report
- **Materialize Action Plan** (tab Execution) — converts JSON into real Salesforce **Tasks + Events**

### 🤖 Automation — Auto-create on Event

A record-triggered flow `Auto_Create_Visit_Capture_On_Event` automatically creates a `Visit_Capture__c` whenever a user books an Event with an `AccountId`. Account/Contact/Opportunity are inherited. Result: **every meeting in the agenda becomes a visit to finalize** in the cockpit.

### 🖥️ LWC experiences

| Component | Surface | Purpose |
|---|---|---|
| `vcVisitWorkspace` | Visit_Capture record page | Immersive 3-tab workspace (Preparation → Execution → Outcome) with AI actions |
| `vcCockpit` + 5 children | `/lightning/n/Visit_Cockpit` | Single-page dashboard: 6 KPIs, 2 action queues, analytics, recent visits, side drawer detail |
| `vcEventVisitCard` | Event record page (Related tab) | Inline Demo Context summary + AI Visit Brief for the visit linked to the Event — no extra click |

All components are **fully bilingual** (English by default, French when the user's Language is set to French) via Custom Labels + translations.

## 📋 Prerequisites

### Einstein Generative AI
- Einstein GenAI turned on
- Licensed edition (Enterprise, Unlimited, or Agentforce 1)
- Model available: **`sfdc_ai__DefaultGPT4Omni`** (used by the 3 prompts). On orgs without GPT-4 Omni, swap the model manually in Prompt Builder after deployment.

### Einstein Conversation Intelligence / In-Person Meeting (optional — for ECI transcript path)

> Full ECI path is only required if you want automatic transcripts from mobile meetings. Without ECI, the component B still works with manual voice transcript (dictation).

**Licenses & editions**: Enterprise (EE), Unlimited (UE), Agentforce 1 (A1E). **Conversation Intelligence** must be licensed and enabled.

**Devices**:
| Platform | Minimum | Recommended |
|---|---|---|
| iOS | iOS 16+ | iPhone 15 Pro or newer |
| Android | 8 GB RAM, 6-core CPU, Android 13+ | Pixel 8+, Samsung S23+ |

**App**: sales reps must use **Sales Cloud Mobile** (not the generic Salesforce app).

**Admin setup (4 steps)**:
1. Setup → "Salesforce Go" → **Einstein Conversation Insights** → toggle ON
2. Setup → Permission Sets → assign **"Einstein Conversation Insights"** to mobile users
3. Setup → Salesforce Go → Einstein Conversation Insights → toggle **"In-Person Meetings"** ON
4. Setup → "Sales Cloud Mobile" → enable the app + distribute (App Store / MDM)

**Privacy**: transcription happens 100% on-device, audio is never transmitted, only the text surfaces in Salesforce. GDPR-compatible. **Not** HIPAA-compliant, **not** available on GovCloud.

**Languages** (driven by the device locale): FR, EN, ES, DE, IT, PT, JA, KO, AR, ZH, NL, PL, RU.

For an org **without ECI**, Component B still works in **standalone mode**: the rep pastes the voice transcript manually (or uses native mobile dictation) into `Voice_Transcript__c`. The prompt falls back automatically.

## 📦 What gets deployed

### Custom objects
- **`Demo_Context__c`** — 11 fields (Account_Name, Industry, Meeting_Type, Demo_Goal, Personas, Context_Notes + 5 AI fields)
- **`Visit_Capture__c`** — 14 fields (Demo_Context, VideoCall, Event, Account, Contact, Opportunity + Voice_Transcript, Visit_Summary + 5 AI fields)

### Einstein Prompt Templates (`flex`, GPT-4 Omni)
- **`Prep_Demo_Grounding`** — demo grounding generation (JSON output, 5 keys)
- **`Prepare_Visit_Brief`** — pre-meeting HTML brief
- **`Generate_Visit_Report`** — post-meeting report + action plan (JSON output)

All prompts use the `{!$User.LanguageLocaleKey}` merge field so the output follows the **invoking user's language**.

### Apex classes
- **`DemoGroundingParser`** — invokes `Prep_Demo_Grounding`, parses JSON, writes 5 fields on Demo_Context
- **`VisitBriefGenerator`** — invokes `Prepare_Visit_Brief`, writes HTML into `AI_Visit_Brief__c`
- **`VisitReportGenerator`** — invokes `Generate_Visit_Report`, writes report + plan + language
- **`ActionPlanMaterializer`** — parses the JSON action plan, creates Tasks + Events linked to the visit
- **`VisitCockpitController`** — AuraEnabled controller backing all LWC (getSummary, getMissingReports, getRecentVisits, getDetail, saveField, regenerateReport, materializePlan, generateBrief, getVisitForEvent, createVisitForEvent, searchDemoContexts…)

### Flows
- **`Generate_Demo_Grounding`** (Screen) → invoked by Demo_Context Quick Action
- **`Generate_Visit_Report_Flow`** (Screen) → invoked by Visit_Capture Quick Action
- **`Materialize_Action_Plan`** (Screen) → invoked by Visit_Capture Quick Action
- **`VC_Grounding`** (PromptFlow, ECI only) — data provider that fetches `VideoCallTranscript` linked to the visit's VideoCall and injects it into `Generate_Visit_Report`
- **`Auto_Create_Visit_Capture_On_Event`** (Record-Triggered, After Save on Create) — auto-creates a Visit_Capture when an Event with `AccountId` is posted

### Quick Actions
- `Demo_Context__c.Generate_Grounding` — "Generate Grounding"
- `Visit_Capture__c.Generate_Report` — "Generate Report"
- `Visit_Capture__c.Materialize_Plan` — "Materialize Action Plan"

### Lightning App + Tabs + FlexiPages
- **Custom App**: `Sales Visits` (FR: "Visites commerciales") with 4 tabs: Visit Cockpit, Visit Captures, Demo Contexts, Events
- **FlexiPages**:
  - `Visit_Cockpit_Page` — hosts the single-page cockpit
  - `Visit_Capture_Record_Page` — full-width record page with `vcVisitWorkspace`
  - `Event_Record_Page` — patched with `vcEventVisitCard` in the Related tab (Demo Context summary + AI Visit Brief inline, no click required)

### Internationalization
- **Custom Labels**: ~110 labels, all in English by default, categorized `VisitCapture`
- **Translations** (`fr.translation-meta.xml`): complete French translations for all UI strings
- **Object Translations**: `Demo_Context__c-fr`, `Visit_Capture__c-fr` (field labels, plural names)

### Permission Set
- **`Demo_Context_User`** — grants app visibility, full R/W on both objects, all fields (FLS), all 5 Apex classes, tab visibility

## 🛠️ Deployment

The repo ships **two manifests**:

### Org WITH ECI / In-Person Meeting (full feature set)
```bash
sf project deploy start --target-org <alias> --manifest manifest/package.xml
```

### Org WITHOUT ECI (no VideoCallTranscript available)
```bash
sf project deploy start --target-org <alias> --manifest manifest/package-no-eci.xml
```
Excludes the `VC_Grounding` flow and the `Generate_Visit_Report` prompt (both depend on `VideoCallTranscript`). Component A works fully; Component B works in manual mode (paste transcript into `Voice_Transcript__c`).

### Full source push (local dev)
```bash
sf project deploy start --target-org <alias> --source-dir force-app
```

## ⚙️ Post-deployment steps (mandatory)

### 1. Assign the permission set to your user
```bash
sf org assign permset --name Demo_Context_User --target-org <alias>
```

### 2. Activate the 3 prompt templates
Prompts land in **Draft** after deployment. Activate each:
1. Setup → **Prompt Builder**
2. Open `Prep_Demo_Grounding` → **Activate** (top-right)
3. Repeat for `Prepare_Visit_Brief` and `Generate_Visit_Report`

### 3. (ECI orgs) Verify the `VC_Grounding` flow
The flow deploys as **Active**, but if you see a newer Draft version in Flow Builder, make sure the right one is active.

### 4. Assign the `Sales Visits` app
Grant app visibility either via the permset (already done if step 1 passed) or Profile → Assigned Apps.

### 5. Activate the Visit Capture Workspace page
Setup → Lightning App Builder → open **Visit Capture Workspace** → **Activation** → "Org Default" (or assign to specific app/profile). Without this, opening a Visit Capture still shows the standard layout instead of the 3-tab workspace.

### 6. Activate the patched Event Record Page (**mandatory**)
The package ships an Event Record Page (`Event_Record_Page.flexipage-meta.xml`) that already embeds `vcEventVisitCard` in the Related tab — so the **Demo Context summary + AI Visit Brief show up inline** the moment a sales rep opens an Event, with zero extra click.

But Salesforce never activates a deployed Lightning Record Page automatically. You must do it once:

1. Setup → **Lightning App Builder** → open **Event Record Page**
2. Click **Activation…** → **Org Default** (or assign to the `Sales Visits` app + the relevant profiles)
3. **Save**
4. **Mobile:** if your org uses a separate **Phone** form-factor view, switch the App Builder canvas to it and confirm `vcEventVisitCard` is present (the bundle declares `Small` + `Large` form factors). Drop it manually if missing, then save and activate the Phone view.

Without this step, opening an Event shows the standard layout and the visit preparation card stays invisible.

### 7. Seed demo data (Claude Code only)
Demo dataset generation lives **outside** the deployable package, under
`scripts/demo-data/`. To spawn realistic prospects, contacts, events,
Demo Contexts and Visit Captures for a live demo, run Claude Code in
this repo and ask:

> *seed demo data for `<seller>`*

Claude will then:

1. **Interview you** (4 questions max): the seller name, what it sells,
   target industries, meeting type and number of scenarios.
2. **Anchor on the seller** via WebSearch (real market customers, sector
   vocabulary, current concerns). Falls back to general knowledge if
   WebSearch is unavailable.
3. **Write a one-shot Apex anonymous file** under
   `scripts/demo-data/_runs/<timestamp>-<seller>.apex` with bespoke,
   French-language grounding fields and visit briefs **written directly
   by Claude** — no Prompt Builder runtime, no Models API quota burnt.
4. **Run it** via `sf apex run --target-org <alias> --file …` and report
   the created Ids.

Each scenario produces: 1 customer Account, 2 named Contacts (with
French names and industry-coherent titles), 1 Demo Context (with
`Company__c` = seller, `Account_Name__c` = customer + the 5 grounding
fields filled in), 1 future Event, 1 Visit Capture with an HTML brief.

#### Example

Asking *"seed demo data for Sopra HR — QBR × 3 — they sell HR software
and want to upsell their new HR Portal"* yields three QBR scenarios at
plausible Sopra HR customers (a manufacturer, a bank, a public-sector
agency), each with sector-specific pain points (e.g. ACPR audit for the
bank, plein-emploi reconversions for the public agency) and an upsell
angle baked into the visit brief.

#### Why outside the package

- **No Prompt Builder runtime dependency** — bypasses the PR0014 errors
  that hit metadata-deployed templates.
- **Per-seller customisation** — every demo gets real market customers
  and sector-accurate language, not a hardcoded pool.
- **Distributable package stays clean** — no admin tooling for end
  customers to wonder about.

See `scripts/demo-data/README.md` for the full workflow and cleanup
guidance.

### 8. (Optional) Switch a user to French to test i18n
Setup → Users → User → **Language = French** → logout/login → all UI strings and AI outputs switch to French seamlessly.

## 📖 User guide

### Workflow A — Prepare a demo
1. Open a **Demo Context** record (or create one)
2. Fill: account name, industry, meeting type, goal, personas, notes
3. Click **"Generate Grounding"** → the 5 AI fields populate in ~10 seconds
4. Review and adjust if needed

### Workflow B — Capture a visit (3 tabs)

Open a Visit Capture. The `vcVisitWorkspace` LWC surfaces 3 tabs:

**Tab 1 — Preparation**
- Pick the linked Demo Context (grounding pre-fills the AI context)
- Confirm Account / Contact / Opportunity
- Click **"Generate visit brief"** → a polished HTML brief appears (context, talking points, opening hooks, objections, success criteria)

**Tab 2 — Execution**
- Link the VideoCall (if ECI is used) — the transcript appears read-only
- Paste manual voice transcript and/or add free notes
- Upload attachments (jpg/png/pdf, max 5 MB each)
- Click **"Generate Report"** → report + plan are generated and you auto-land on tab 3
- Click **"Materialize Action Plan"** → real Tasks + Events appear in the CRM

**Tab 3 — Outcome**
- Reads the HTML report, the action plan as cards, and the files
- Everything stays persistent — reopening the record shows the latest state

### Workflow C — Cockpit overview
Navigate to `/lightning/n/Visit_Cockpit` (or from the app). The cockpit shows:
- **6 KPIs** with deltas vs previous period (total visits, to-finalize, pending plans, tasks+events, ECI captured, manual captured)
- **2 action queues**: visits missing reports, plans pending materialization
- **Analytics**: weekly volume sparkline, source breakdown donut, top 5 accounts bar chart
- **Recent visits** table (last 10) with clickable rows opening the drawer
- **Side drawer** for quick report review / field autosave / AI regeneration without leaving the cockpit

## 🧠 Design decisions

- **Rich Text Area** for `AI_Visit_Brief__c` and `AI_Report__c` — the HTML renders natively in the standard UI, reports, and mobile app. No dedicated LWC needed for the detail page.
- **JSON sortie for plans** — `AI_Action_Plan__c` keeps a structured payload so `ActionPlanMaterializer` can re-parse and create typed records (tasks vs events with priority, duration, due dates).
- **`{!$User.LanguageLocaleKey}` merge field** — used in all 3 prompts to drive output language from the invoking user's locale, without needing `outputLanguageMode` (not yet fully available via Metadata API in all orgs).
- **Text(18) + Formula Hyperlink** for `Event__c` / `Event_Link__c` — Salesforce disallows custom lookups pointing at `Event`, so the field holds the Id as text and a formula provides the clickable link.
- **LWC cockpit, not a report** — the cockpit is a single-page LWC (no standard dashboard) because we need interactive drawers, autosave, AI re-generation, and custom visuals (donut, sparkline) in one immersive canvas.
